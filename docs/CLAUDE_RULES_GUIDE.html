1. What's New

Claude Code now supports modular rules via the .claude/rules/ directory. Instead of maintaining one large CLAUDE.md file, you can split your instructions into multiple focused markdown files.

  • All .md files in .claude/rules/ are automatically loaded
  • Rules can be scoped to specific file paths using YAML frontmatter
  • Supports subdirectories for organization
  • Symlinks supported for sharing rules across projects
  • User-level rules in ~/.claude/rules/ apply globally

2. Side-by-Side Comparison

CLAUDE.md (Traditional)

your-project/
├── CLAUDE.md     # Everything in one file
└── src/
    └── ...

All project instructions, coding standards, API rules, and testing guidelines in a single file.

.claude/rules/ (Modular) NEW

your-project/
├── .claude/
│   ├── CLAUDE.md      # High-level info
│   └── rules/
│       ├── code-style.md
│       ├── testing.md
│       └── security.md
└── src/

Split by topic. Path-specific rules reduce context overhead.

3. Key Features

Path-Specific Rules (The Killer Feature)

Rules can be scoped to only apply when working on certain files. Claude only sees relevant rules, reducing context overhead.

Example: API-specific rules 
---
paths: src/api/**/*.ts
---

# API Development Rules

- All API endpoints must include input validation
- Use the standard error response format
- Include OpenAPI documentation comments

These rules only load when working on files matching src/api/**/*.ts.

Glob Pattern Support

Pattern Matches
**/*.ts All TypeScript files in any directory
src/**/* All files under src/ directory
*.md Markdown files in project root
src/components/*.tsx React components in specific directory
src/**/*.{ts,tsx} TypeScript and TSX files under src/
{src,lib}/**/*.ts TS files in multiple directories

User-Level Rules

Create personal rules in ~/.claude/rules/ that apply to all your projects:

~/.claude/rules/
├── preferences.md    # Personal coding preferences
└── workflows.md      # Preferred workflows
Priority: User-level rules load first, then project rules. Project rules have higher priority (can override user rules).

Symlinks for Shared Rules

Share common rules across projects using symlinks:

# Symlink a shared rules directory
ln -s ~/shared-claude-rules .claude/rules/shared

# Symlink individual rule files
ln -s ~/company-standards/security.md .claude/rules/security.md

4. Recommended Directory Structure

.claude/
├── CLAUDE.md              # High-level project info, tech stack
└── rules/
    ├── code-style.md      # Formatting, naming conventions
    ├── testing.md         # Test patterns, coverage requirements
    ├── security.md        # Security practices (all files)
    ├── frontend/
    │   ├── react.md       # paths: src/components/**
    │   └── styles.md      # paths: **/*.css, **/*.scss
    └── backend/
        ├── api.md         # paths: src/api/**
        └── database.md    # paths: prisma/**, src/db/**

5. Path-Specific Rules Examples

React Components

---
paths: src/components/**/*.tsx
---

# React Rules

- Use functional components
- Props interface above component
- Export as default
- Co-locate styles and tests

API Routes

---
paths: src/api/**/*.ts
---

# API Rules

- Validate input with Zod
- Consistent error format
- Log errors with context
- Rate limit public endpoints

Database Layer

---
paths: prisma/**, src/db/**
---

# Database Rules

- Use transactions for writes
- Index queried fields
- Soft delete over hard delete
- Use migrations, not db push

Test Files

---
paths: **/*.test.ts, **/*.spec.ts
---

# Testing Rules

- Arrange-Act-Assert pattern
- One assertion per test
- Mock external dependencies
- Descriptive test names

6. Should You Migrate?

Yes, migrate if:

  • CLAUDE.md is over 200 lines
  • Rules only apply to certain file types
  • Multiple people edit project instructions
  • You want to share rules across projects
  • Different codebase parts need different rules

No need if:

  • CLAUDE.md is short and focused
  • All rules apply project-wide
  • Single maintainer
  • Simple project structure
Good news: You can use both! They load with the same priority. Start by extracting specific topic areas into rules files while keeping high-level info in CLAUDE.md.

7. Best Practices

  • Keep rules focused: Each file covers one topic (e.g., testing.md, api-design.md)
  • Use descriptive filenames: Name indicates what rules cover at a glance
  • Use conditional rules sparingly: Only add paths when rules truly apply to specific files
  • Organize with subdirectories: Group related rules (e.g., frontend/, backend/)
  • Keep CLAUDE.md for overview: Tech stack, project context, universal conventions
  • Review rule loading: Use /memory to see which rules are loaded
Key benefit: Reduced context overhead — Claude only loads rules relevant to the files you're working on.
